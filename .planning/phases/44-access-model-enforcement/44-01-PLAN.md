<audit>

## Audit — enforcement gaps in the current codebase

Grepped `requireAdmin`, `getAuthenticatedUser`, `canAccessProject`, `roleAtLeast`, `ownerId`, `user.role`, `collaborators?.some`, `allowComments`, `allowDownloads`, `reviewLinkId` across `src/app/api/**`.

### Platform role gaps
- `roleAtLeast` is only called in `folders POST` (needs manager) and `review-links POST` (needs manager). It is NOT called in `upload/signed-url`, `upload/complete`, `upload/thumbnail`, `assets/copy`, `assets/[assetId] PUT/DELETE`, `assets/merge-version`, `assets/unstack-version`, `assets/reorder-versions`, `folders PUT`, `folders/copy` — so platform role `viewer` can upload, rename assets, mutate stacks, duplicate folders, etc.
- `folders POST` and `review-links POST` check PLATFORM role but ignore PROJECT role: a platform `editor` who is a project `owner` cannot create folders or review links in their own project (false negative).
- `projects GET` does not admit platform admin override (inconsistent with DELETE which does).

### Project role gaps
- `projects/[projectId] PUT` (rename/description): only owner matches via `ownerId` — platform admin cannot rename, project-level `owner` from collaborators array (set-based membership) cannot edit.
- `projects/[projectId]/collaborators POST|DELETE`: only `ownerId` — platform admin cannot manage collaborators (except via `/api/admin/users/[userId]/project-access`).
- `assets/[assetId] PUT/DELETE`: only checks `canAccessProject` — project `reviewer` role can rename and DELETE assets.
- `assets/copy`, `assets/merge-version`, `assets/unstack-version`, `assets/reorder-versions`: same — reviewers can mutate version stacks.
- `folders/[folderId] PUT`: only `canAccessProject` — reviewers can rename/move folders.
- `folders/[folderId] DELETE`: checks platform `roleAtLeast(user, 'manager')` — ignores project role, so a platform-editor who owns the project cannot delete their own folders.
- `folders/copy`: only project access — reviewers can duplicate folders.
- `review-links/[token] PATCH|DELETE`: only `createdBy === uid` — project owner/admin cannot revoke or rename a review link created by a former collaborator.
- `comments POST` (internal auth path): only project access — viewer/reviewer roles are not distinguished.
- `comments/[commentId] PUT/DELETE`: author-or-admin check is OK but misses project owner moderation.
- `upload/complete`: only `uploadedBy === user.id` — legitimate if constrained to the signed-url creator, but combined with upload/signed-url having no role check it means any project collaborator (including reviewer) can call upload/signed-url then upload/complete.

### Review-link permission flag gaps
- `allowComments`: NOT enforced server-side in `comments POST`. A guest with a valid `reviewLinkId` where `link.allowComments === false` can still POST comments. Critical hole.
- `expiresAt`: checked ONLY in `review-links/[token] GET`. `comments POST` guest path, `comments GET ?reviewToken=`, `review-links/[token]/contents` do NOT check expiry — expired-link guests can still read/write comments.
- `password`: checked only on `review-links/[token] GET`. Guest does not re-present the password on `comments POST` — server has no way to enforce it for subsequent writes (design gap; mitigate by requiring password handshake to produce a short-lived review-link-scoped token, OR re-validate password from query/header on every guest write). For v1.7 we accept the simpler route: require `password` param on every guest write to the same link.
- `allowDownloads`: correctly gated on `review-links/[token] GET` decoration. OK.
- `allowApprovals`: referenced in UI only — no server endpoint exists yet for approval. Plan includes a stub: if/when comments POST includes `approvalStatus` from review-link auth, it must reject when `allowApprovals === false`.
- `showAllVersions`: correctly gated on `review-links/[token] GET` grouping. OK.
- `comments GET ?reviewToken=`: does not check expiry or password. A guest with a stale token can still read old comments.

### Silent fallthrough risks
- `projects/[projectId] GET` has no admin override (vs DELETE which does) — not a security hole but inconsistent.
- `comments/[commentId] PUT` returns 403 "No updatable fields" when user has project access but no allowed fields in payload — OK, explicit.
- All routes use try/catch that swallow errors as `500` — fine, but `console.error` is inconsistent; not in scope.

</audit>

<design>

## Permission module design — `src/lib/permissions.ts`

Single file, no runtime Firestore access (pure functions — callers pass already-fetched `User`, `Project`, `ReviewLink`, `Comment`). Separate module `src/lib/permissions-db.ts` MAY exist for DB-aware wrappers (e.g., `loadProject` + `canDeleteProject`) but keep the pure matrix testable in isolation.

### Platform role ranking
```ts
export const PLATFORM_ROLE_RANK = { viewer: 0, editor: 1, manager: 2, admin: 3 } as const;
export type PlatformRole = keyof typeof PLATFORM_ROLE_RANK;
export function platformRoleAtLeast(user: User, min: PlatformRole): boolean;
```

### Project role
```ts
export type ProjectRole = 'owner' | 'editor' | 'reviewer';

/** Admin override: always returns 'owner' for platform admins. Otherwise the collaborator entry's role, or null. */
export function getProjectRole(user: User, project: Project): ProjectRole | null;

/** Semantic, public. */
export function canAccessProject(user: User, project: Project): boolean;
```

### Pure action checks (signatures)
Each returns `boolean`. No side effects. No throws.

Project scope:
- `canRenameProject(u, p)` — platform admin OR project owner
- `canDeleteProject(u, p)` — platform admin OR project owner
- `canInviteCollaborator(u, p)` — platform admin OR project owner
- `canRemoveCollaborator(u, p)` — platform admin OR project owner

Asset/stack scope (writes):
- `canUpload(u, p)` — platform role >= editor AND (project role in {owner, editor} OR platform admin)
- `canDeleteAsset(u, p)` — platform admin OR project role in {owner, editor}
- `canRenameAsset(u, p)` — same as canDeleteAsset
- `canCopyAsset(u, p)` — same as canUpload
- `canModifyStack(u, p)` — same as canDeleteAsset (covers merge/unstack/reorder)
- `canProbeAsset(u, p)` / `canGenerateSprite(u, p)` — same as canUpload (they mutate stored metadata)

Folder scope:
- `canCreateFolder(u, p)` — same as canUpload
- `canRenameFolder(u, p)` — same as canDeleteAsset
- `canDeleteFolder(u, p)` — same as canDeleteAsset

Review-link scope:
- `canCreateReviewLink(u, p)` — platform role >= editor AND (project role in {owner, editor} OR platform admin)
- `canEditReviewLink(u, p, link)` — platform admin OR project owner OR link.createdBy === u.id
- `canDeleteReviewLink(u, p, link)` — same as canEditReviewLink

Comments scope (authenticated):
- `canPostComment(u, p)` — any project role (including reviewer); platform role >= viewer
- `canResolveComment(u, p)` — any project role
- `canEditComment(u, p, c)` — platform admin OR (c.authorId === u.id AND project access)
- `canDeleteComment(u, p, c)` — platform admin OR project owner OR (c.authorId === u.id)

```ts
  // throws ReviewLinkDenied('comments_disabled') if action === 'comment' && !link.allowComments
  // throws ReviewLinkDenied('approvals_disabled') if action === 'approve' && !link.allowApprovals

```

Routes catch `ReviewLinkDenied` and map each reason to a 401/403/410 with a clear message.

### Route helper pattern
Each refactored route does:
```ts
const user = await getAuthenticatedUser(request);
if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

const project = await loadProject(params.projectId); // DB read
if (!project) return NextResponse.json({ error: 'Not found' }, { status: 404 });

if (!canDeleteProject(user, project)) {
  return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
}
// …proceed
```

### Migration strategy for auth-helpers
- `roleAtLeast` and `canAccessProject` move to permissions.ts.
- `auth-helpers.ts` re-exports `canAccessProject` from permissions as a thin async wrapper `canAccessProjectById(userId, projectId)` that loads from Firestore — ONLY used by legacy code paths until the per-task refactors replace them all. By end of Task 9, `auth-helpers.ts` contains only `verifyAuthToken`, `getAuthenticatedUser`, `requireAdmin`, `getIdTokenFromRequest`.

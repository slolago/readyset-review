# Plan 1: Fix review link "not found" on fresh links

## Objective

Newly created review links return a 404 "not found" error when the review page tries to
load them immediately after creation. Fix the lookup so a freshly created link is
accessible with no delay.

## Root Cause Analysis

The POST handler in `src/app/api/review-links/route.ts` stores the review link in
Firestore as a document with a randomly generated `token` field (not used as the doc ID).
The GET handler in `src/app/api/review-links/[token]/route.ts` looks up that document
with a collection query: `.where('token', '==', params.token).limit(1).get()`.

This query requires a Firestore single-field or composite index on `reviewLinks.token`.
Without a deployed index, Firestore either (a) throws an error that is swallowed by the
catch block returning 500, or (b) returns an empty result set, causing `snap.empty` to be
`true`, which returns a 404. The POST API response includes the `token` value in
`data.link.token`, so the token itself is correct — the lookup just fails.

The fix: use the token as the Firestore document ID instead of an auto-generated ID. Then
the GET handler can use `db.collection('reviewLinks').doc(token).get()` — a direct
document fetch that needs no index and has no eventual-consistency delay.

## Context

### Files involved

- `src/app/api/review-links/route.ts` — POST creates the document
- `src/app/api/review-links/[token]/route.ts` — GET/DELETE looks up by token
- `src/app/review/[token]/page.tsx` — client that calls GET (no changes needed)
- `src/components/review/CreateReviewLinkModal.tsx` — calls POST (no changes needed)

### Current document structure in Firestore (reviewLinks collection)

```
{
  id: <auto-generated Firestore doc ID>,
  token: <32-char random string>,   ← used for URL /review/[token]
  name, projectId, folderId, createdBy, allowComments, expiresAt, createdAt, password?
}
```

### Target document structure

```
{
  id: <token>,   ← token IS the doc ID, no separate token field needed in queries
  token: <same value, kept for backward compatibility with existing docs>,
  ...same fields
}
```

## Tasks

### Task 1: Update POST to use token as document ID

File: `src/app/api/review-links/route.ts`

1. Keep `const token = generateToken()` as-is.
2. Replace `await db.collection('reviewLinks').add(data)` with
   `await db.collection('reviewLinks').doc(token).set(data)`.
3. Replace `const doc = await ref.get()` / `return { id: ref.id, ...doc.data() }` with
   a direct read: `const doc = await db.collection('reviewLinks').doc(token).get()` and
   return `{ link: { id: token, ...doc.data() } }`.
4. The `data` object already includes `token` as a field — keep it so existing documents
   and any future queries that still use `.where('token', ...)` continue to work.

The updated POST function body (after validation and auth checks) should be:

```typescript
const token = generateToken();
const db = getAdminDb();

const data: Record<string, unknown> = {
  token,
  name,
  projectId,
  folderId: folderId || null,
  createdBy: user.id,
  allowComments: allowComments !== false,
  expiresAt: expiresAt ? Timestamp.fromDate(new Date(expiresAt)) : null,
  createdAt: Timestamp.now(),
};
if (password) data.password = password;

await db.collection('reviewLinks').doc(token).set(data);
const doc = await db.collection('reviewLinks').doc(token).get();

return NextResponse.json({ link: { id: token, ...doc.data() } }, { status: 201 });
```

### Task 2: Update GET and DELETE to use doc(token) directly

File: `src/app/api/review-links/[token]/route.ts`

Replace the collection query in both GET and DELETE with a direct document fetch:

**GET** — replace:
```typescript
const snap = await db.collection('reviewLinks')
  .where('token', '==', params.token)
  .limit(1)
  .get();

if (snap.empty) return NextResponse.json({ error: 'Not found' }, { status: 404 });

const doc = snap.docs[0];
const link = { id: doc.id, ...doc.data() } as any;
```

with:
```typescript
const doc = await db.collection('reviewLinks').doc(params.token).get();

if (!doc.exists) return NextResponse.json({ error: 'Not found' }, { status: 404 });

const link = { id: doc.id, ...doc.data() } as any;
```

**DELETE** — replace:
```typescript
const snap = await db.collection('reviewLinks')
  .where('token', '==', params.token)
  .limit(1)
  .get();

if (snap.empty) return NextResponse.json({ error: 'Not found' }, { status: 404 });

const link = snap.docs[0].data() as any;
if (link.createdBy !== decoded.uid) { ... }

await snap.docs[0].ref.delete();
```

with:
```typescript
const doc = await db.collection('reviewLinks').doc(params.token).get();

if (!doc.exists) return NextResponse.json({ error: 'Not found' }, { status: 404 });

const link = doc.data() as any;
if (link.createdBy !== decoded.uid) {
  return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
}

await doc.ref.delete();
```

### Task 3: Update GET /api/review-links (list endpoint) to return correct IDs

File: `src/app/api/review-links/route.ts` (GET handler)

The list query uses `.where('projectId', '==', projectId)` which still works. The `id` in
the response currently comes from `d.id` (the Firestore doc ID). After the change, new
docs have `id === token`. Existing docs have auto-generated IDs — those will continue to
work for display but their GET-by-token will still use the query path and may fail.

To handle the transition gracefully: update the GET list handler to check `d.data().token`
and use it as a fallback identifier in the response. No structural change needed — the
`.where` query on `projectId` does not require a `token` index. Leave as-is.

No changes needed to the list GET handler.

## Acceptance Criteria

1. After clicking "Create Link" in the modal, the new link URL (`/review/<token>`) loads
   successfully immediately — no "Link not found or expired" error, no 404.
2. Clicking the external link icon in the modal opens the review page with content.
3. The DELETE action from the review links list still works (deletes the correct document).
4. Existing review links created before this fix (with auto-generated doc IDs) continue to
   work — they still have `token` stored as a field and the GET handler now uses
   `doc(token)` which looks up by ID. Old docs won't match by ID, but they are
   backward-compatible: the `.where('token', ...)` query approach is no longer used, so
   old links will 404. This is acceptable — old broken links were already broken.
   If backward compatibility for old links is required, add a fallback query after the
   `!doc.exists` check (see note below).

### Optional backward-compatibility fallback (add if needed)

In `GET /api/review-links/[token]/route.ts`, after `if (!doc.exists)`:

```typescript
// Fallback for legacy documents created before token-as-ID migration
const fallbackSnap = await db.collection('reviewLinks')
  .where('token', '==', params.token)
  .limit(1)
  .get();
if (fallbackSnap.empty) return NextResponse.json({ error: 'Not found' }, { status: 404 });
const fallbackDoc = fallbackSnap.docs[0];
const link = { id: fallbackDoc.id, ...fallbackDoc.data() } as any;
// ... continue with rest of handler using `link`
```

This requires a Firestore index on `reviewLinks.token` for the fallback path only. Since
no new docs will use this path, it can be added as a low-priority index or skipped.

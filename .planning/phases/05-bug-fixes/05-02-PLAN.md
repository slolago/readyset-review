# Plan 2: Fix upload stuck at "Uploading..." for subsequent uploads

## Objective

After uploading a first video to a folder, subsequent uploads to the same folder appear
stuck at "Uploading..." and never transition to complete. Fix the upload state management
so every upload — regardless of order or navigation — finishes and the asset appears in
the folder.

## Root Cause Analysis

There are two compounding issues:

### Issue A: `clearCompleted` wipes in-progress uploads

In `src/hooks/useAssets.ts`, `clearCompleted` is:
```typescript
const clearCompleted = () => {
  setUploads([]);
};
```

This sets `uploads` to an empty array unconditionally. If the user clicks "Dismiss" while
an upload is in progress, the entry for that upload is removed from state. Subsequent
`updateUpload(uploadId, ...)` calls use `prev.map(u => u.id === id ? ... : u)` — they
silently find no matching entry and return the array unchanged. The XHR itself continues
running and the `/api/upload/complete` call succeeds, but the UI never reflects completion
and `refetchAssets()` is never triggered for uploads that finished after the wipe.

Fix: only remove uploads that are in a terminal state (`complete` or `error`).

### Issue B: FolderBrowser re-mount loses in-flight upload tracking

`useUpload` is instantiated inside `FolderBrowser`. When the user navigates to a
subfolder, `FolderBrowser` unmounts. Any XHR still running has a closure over the OLD
`setUploads` from the unmounted component instance. React discards state updates from
unmounted components (with a warning in development). The XHR finishes and calls
`/api/upload/complete` — the asset IS saved as `ready` in Firestore — but `refetchAssets`
is never called because the upload state machinery is orphaned. The new `FolderBrowser`
instance starts with `uploads = []` and has no knowledge of the completed upload.

Fix: after uploading, always call `refetchAssets()` when the upload result returns an
`assetId`, regardless of whether the upload-state update succeeded. The key is: the
`uploadFile` function returns a Promise that resolves to `assetId | null`. The callers
in `FolderBrowser` already use `results.some(r => r !== null)` to trigger `refetchAssets`.
The issue is that when `uploads` state is wiped, `updateUpload` with `status: 'complete'`
is a no-op, but `uploadFile` still returns the `assetId`. So `refetchAssets` IS called —
but on the old (unmounted) component instance.

The real fix for Issue B: move `refetchAssets` into `uploadFile` itself so it fires
regardless of component lifecycle, OR accept the current behavior and explain it is not
actually broken (the asset DOES appear after the next manual navigation or refresh).

After closer reading: the callers `handleFileInputChange`, `handleDrop`, and
`handleFolderInputChange` all `await Promise.all(files.map(f => uploadFile(...)))` and
then call `refetchAssets()`. This `await` will resolve when all uploads finish — including
when the component is still mounted. Navigation during an upload is the edge case.

**Primary fix (confirmed): Issue A is the definitive "stuck" bug.** When a user uploads
two files rapidly, the first completes and the "Dismiss" button appears in the upload
panel. If the user clicks Dismiss before the second upload finishes, the second upload's
entry is wiped, its `updateUpload` calls become no-ops, and `uploadFile` catches no error
(the XHR still resolves, complete API still succeeds) but `updateUpload(uploadId, { status: 'complete' })` is a silent no-op so the caller's `results` still contains the
`assetId`. Wait — `uploadFile` still returns `assetId` even after the wipe, so
`refetchAssets` would be called...

Actually: let me re-read. After `clearCompleted()` wipes uploads, the second upload's
XHR is still running. `updateUpload` is a no-op but doesn't throw. `xhr.addEventListener('load', ...)` fires, `resolve()` is called, execution continues to
Step 3 (`/api/upload/complete`), which succeeds, `updateUpload(uploadId, { status: 'complete', progress: 100 })` is a no-op, and `uploadFile` returns `assetId`. The caller
does `results.some(r => r !== null)` → true → calls `refetchAssets()`.

So Issue A doesn't prevent `refetchAssets` — it only breaks the progress UI display.

**The actual stuck bug:** The progress panel shows "Uploading..." and never visually
updates to "Complete" because after `clearCompleted` wipes the entry, the entry is gone
from `uploads`, so the panel with `uploads.length > 0` disappears entirely. When the
second upload truly finishes and its new entry (added at the start of `uploadFile`) is
updated to `complete`, the panel reappears — but only if the entry survived. If it was
wiped, no panel, no visual feedback.

However the bug report says the upload "never completes" — meaning the asset does not
appear in the folder either. This points to a scenario where `uploadFile` itself throws or
the GCS PUT fails for the second upload. The most likely cause of that:

**Issue C: GCS signed URL or XHR failure on second upload**

The signed URL is generated per-upload with a fresh call to `/api/upload/signed-url`. Each
URL is unique. The XHR PUT to GCS is independent. The `Content-Type` header is set on the
XHR request. This should work.

**Issue D: Thumbnail capture hangs for 15 seconds on second video**

`captureThumbnail` has a 15-second `setTimeout` fallback. If the second video's blob URL
causes the `<video>` element to hang (browser resource limits, second video element while
first may still be partially alive), thumbnail capture blocks for 15 seconds before
continuing. During this 15-second wait, the upload panel shows "Uploading..." with 0%
progress, which looks "stuck." After 15 seconds, upload proceeds normally.

This is very likely the actual user-visible bug: the second upload appears stuck for up
to 15 seconds because `captureThumbnail` is blocking the rest of `uploadFile`.

**Fix for Issue D:** Run thumbnail capture with a shorter timeout (5 seconds instead of
15), and ensure the video element is fully cleaned up after capture.

**Fix summary:**
1. Fix `clearCompleted` to only remove terminal uploads (clean UX, prevents phantom state)
2. Reduce `captureThumbnail` timeout from 15 seconds to 5 seconds (eliminates long stall)
3. Ensure `video.src = ''` and `URL.revokeObjectURL` always run on settle (already done
   via `done()` — verify the `settled` guard works correctly for the timeout path)

## Tasks

### Task 1: Fix `clearCompleted` to only remove terminal uploads

File: `src/hooks/useAssets.ts`

Replace:
```typescript
const clearCompleted = () => {
  setUploads([]);
};
```

With:
```typescript
const clearCompleted = () => {
  setUploads((prev) => prev.filter((u) => u.status === 'uploading' || u.status === 'pending'));
};
```

This keeps in-progress and pending uploads in the list. Clicking "Dismiss" only removes
uploads that have reached `complete` or `error` status.

Also rename the button label in `FolderBrowser.tsx` from "Dismiss" to "Clear completed"
to accurately reflect what it does:

File: `src/components/files/FolderBrowser.tsx` (line ~728)

Change:
```tsx
<button onClick={clearCompleted} className="...">
  Dismiss
</button>
```

To:
```tsx
<button onClick={clearCompleted} className="...">
  Clear completed
</button>
```

### Task 2: Reduce thumbnail capture timeout and verify cleanup

File: `src/hooks/useAssets.ts`

Change the timeout in `captureThumbnail` from 15 seconds to 5 seconds:

```typescript
// Before
setTimeout(() => done(null), 15000);

// After
setTimeout(() => done(null), 5000);
```

Also verify the `done()` path on timeout calls `video.src = ''` and
`URL.revokeObjectURL(url)`. The current `done` function does:
```typescript
const done = (blob: Blob | null) => {
  if (settled) return;
  settled = true;
  video.src = '';
  URL.revokeObjectURL(url);
  resolve(blob);
};
```

The `settled` guard is correct — `done` is idempotent. The timeout calls `done(null)`,
which clears `video.src` and revokes the URL. No change needed here beyond the timeout
duration.

### Task 3: Add error logging to XHR upload for easier future diagnosis

File: `src/hooks/useAssets.ts`

The XHR `error` event currently logs to console but could be more informative. Verify
the existing logging is adequate and add the upload filename to the log message:

In `uploadFile`, after `const { signedUrl, assetId } = await signedRes.json()`, the
`file.name` is available in scope. Update the XHR error log:

```typescript
xhr.addEventListener('error', (e) => {
  console.error(`[upload] GCS network error for "${file.name}":`, e);
  reject(new Error('Network error — check CORS config on GCS bucket'));
});
```

And the load failure log:
```typescript
if (xhr.status >= 200 && xhr.status < 300) {
  resolve();
} else {
  console.error(`[upload] GCS upload failed for "${file.name}":`, xhr.status, xhr.responseText);
  reject(new Error(`GCS upload failed (${xhr.status}): ${xhr.responseText || 'No response'}`));
}
```

This makes it much easier to diagnose any remaining upload issues in production logs.

## Acceptance Criteria

1. Uploading a first video to a folder completes and the asset appears in the grid.
2. Uploading a second video to the same folder immediately after the first completes
   successfully — the asset appears in the grid within a few seconds of the upload
   finishing.
3. The second upload does not appear stuck at "Uploading..." for more than 5 seconds
   while thumbnail capture runs.
4. Clicking "Clear completed" while a second upload is in progress removes only the
   completed entries from the upload panel — the in-progress entry remains visible.
5. After all uploads finish, clicking "Clear completed" dismisses the panel entirely.

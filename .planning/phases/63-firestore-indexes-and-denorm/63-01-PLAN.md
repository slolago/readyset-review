# Phase 63 Plan 01: Firestore Indexes & Denormalization

## Objective

Eliminate hot-path full-collection scans. Replace fetch-then-filter in `/api/assets`, `/api/folders`, and the trash endpoint with indexed composite queries. Denormalize `commentCount` onto the asset doc so the list endpoint no longer scans the `comments` collection.

## Tasks

1. Create `firestore.indexes.json` with composite indexes for assets(projectId, folderId, deletedAt), folders(projectId, parentId, deletedAt), comments(assetId, parentId, createdAt), plus (projectId, deletedAt) for trash on both assets & folders.
2. Add `commentCount: number` to `Asset` type; initialize to 0 on asset create; wrap comment POST + DELETE in a transaction that increments/decrements `asset.commentCount`; replace the comment-count scan in `/api/assets` with a read of `asset.commentCount` (lazy backfill on first read if missing).
3. Switch `/api/assets` to an indexed `(projectId, folderId, deletedAt)` query, with a FAILED_PRECONDITION fallback to the legacy in-memory filter.
4. Switch `/api/folders` to an indexed `(projectId, parentId, deletedAt)` query, same fallback pattern.
5. Switch `/api/projects/[id]/trash` to indexed `(projectId, deletedAt)` queries for both assets and folders.

## Verification

- `tsc --noEmit` clean
- `vitest run` clean
- Every task committed separately; no push.

## Anti-Scope

- No data-migration script — `commentCount` is lazy-backfilled on first list query per asset.
- No admin UI for index deploy status.
- No schema changes beyond `commentCount`.

<objective>
Fix FS-01: folder Duplicate actually persists a visible copy. Root cause: `deepCopyFolder` creates Firestore folder docs without a `deletedAt` field; the listing query (`GET /api/folders`, `src/app/api/folders/route.ts:54`) uses the Phase 63 composite index `where('deletedAt', '==', null)` which EXCLUDES docs missing the field. So the duplicate is created in Firestore (API returns 201, toast fires) but never appears in the current listing — matching the "fires a toast on nothing" bug report.

Purpose: Restore parity with asset duplicate (which correctly writes `deletedAt: null` at `src/app/api/assets/copy/route.ts:87`) and with regular folder creation (which does so at `src/app/api/folders/route.ts:118`). Also silently repairs "Copy to folder" which has the identical bug through the same code path.

Output: A two-line fix in `src/lib/folders.ts` that adds `deletedAt: null` to both the root-folder `.set(...)` and the subfolder `.set(...)` inside the BFS loop. No client changes, no new routes — existing `handleDuplicateFolder` at `FolderBrowser.tsx:871-888` already awaits the API and refetches; it only looked broken because the refetch returned a list that (correctly, per its query) excluded the new doc.
</objective>

<scouting-notes>
Traced the reported bug end-to-end:

1. **Client handler** (`src/components/files/FolderBrowser.tsx:871-888`, `handleDuplicateFolder`) is correct:
   - POSTs `/api/folders/copy` with `{ folderId }` (no targetParentId → route defaults to same parent, see route.ts line 37).
   - `await`s the response.
   - On `res.ok` fires `toast.success('Folder duplicated')` and calls `fetchFolders()`.
   - On failure fires error toast.

2. **Route** (`src/app/api/folders/copy/route.ts`) is correct:
   - Permission check, load source doc, compute `destinationParentId` (defaults to `source.parentId` when omitted — i.e. same parent as source = "duplicate in place").
   - Calls `deepCopyFolder(...)`.
   - Reads the new root doc back and returns `{ folder, counts }` with 201.

3. **Helper** (`src/lib/folders.ts:42-48`) creates the new root folder WITHOUT `deletedAt`:
   ```typescript
   await newRootRef.set({
     name: overrideName ?? rootData.name,
     projectId,
     parentId: dstParentId ?? null,
     path: dstRootPath,
     createdAt: Timestamp.now(),
     // ← deletedAt missing
   });
   ```

4. **Same bug in the subfolder loop** at lines 122-129 (children created in BFS also lack `deletedAt`).

5. **Listing query** (`src/app/api/folders/route.ts:50-55`, GET handler) uses the Phase 63 composite index:
   ```typescript
   .where('projectId', '==', projectId)
   .where('parentId', '==', parentId)
   .where('deletedAt', '==', null)
   ```
   Firestore's `== null` matches docs where the field IS explicitly null, NOT docs where the field is absent. So new duplicates, which have no `deletedAt` at all, are excluded.

6. **Precedent for the fix** — both sibling write paths already set `deletedAt: null`:
   - `src/app/api/folders/route.ts:118` (regular folder create)
   - `src/app/api/assets/copy/route.ts:87` (asset duplicate) — with an explanatory comment about Phase 63 indexed queries

7. **Fallback branch** (`src/app/api/folders/route.ts:58-72`) filters `!f.deletedAt` in-memory, which DOES accept absent fields — so on machines where the composite index isn't deployed the bug would not reproduce. Production has the index deployed (per STATE.md line 69), confirming why this manifests in production-style environments.

8. **Naming rule** — `deepCopyFolder` already uses `overrideName ?? rootData.name` (line 43). `handleDuplicateFolder` doesn't pass `name`, so duplicate gets the same name as source. That's parity with asset duplicate (v1.5 Phase 39) — no change needed.

9. **Blast radius** — `deepCopyFolder` has exactly one caller (`/api/folders/copy`), which serves both the "Duplicate" flow AND the "Copy to folder" flow. Both are silently broken in production the same way; both are repaired by the same two-line fix. No existing tests in `src/lib/__tests__` (directory doesn't exist).
</scouting-notes>

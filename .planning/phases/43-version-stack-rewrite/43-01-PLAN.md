<objective>
Audit and fix the version-stack subsystem so every success criterion in Phase 43 is observably true.

<audit_findings>

## Bugs Found in Current Code

### Bug 1 — merge-version: legacy-root drop (silent data loss)
`merge-version/route.ts` queries `where('versionGroupId', '==', sourceGroupId)` then pushes `sourceId` if missing. **But if the user drags a non-root member of a legacy stack (where the root has no `versionGroupId` field), the root is never included in `sourceMembers`** → root stays behind with a dangling `versionGroupId` pointer to the now-empty source group while its siblings migrate to the target. Same issue symmetric on target side, but target side is less destructive (target keeps membership). Root-inclusion guard only protects the case where `sourceId === rootId`.

### Bug 2 — unstack-version: root detach creates ghost group (silent data loss)
When the detached asset IS the original root (`assetId === groupId`), line 75 writes `versionGroupId: assetId` — but remaining members still carry `versionGroupId: groupId` which equals `assetId`. They continue to query-match the "standalone" root. Result: unstacking the root creates a ghost stack where the detached asset appears as a member of its former group on next GET. The remaining members need to be re-rooted to a new groupId (pick `remaining[0].id`).

### Bug 3 — unstack-version: legacy-root missed in members query
Same legacy-root issue as Bug 1. The root-inclusion guard exists (lines 54-60) but uses `db.collection('assets').doc(groupId).get()` — this relies on `groupId` being correctly derived. If asset is a non-root member of a legacy stack (asset has `versionGroupId` pointing to the legacy root, root has no `versionGroupId`), the guard works. Verify this is actually correct under review.

### Bug 4 — reorder-versions: no completeness check (silent partial renumber)
Endpoint verifies all orderedIds share a versionGroupId, but never verifies that `orderedIds.length` equals the full group size. A caller that sends N-1 of N members would successfully renumber those N-1 to `1..N-1`, leaving the missing member with a stale number (potentially colliding). Must assert `orderedIds.length === groupSize`.

### Gap 5 — No context-menu "Stack onto…" affordance
Users can only merge via drag-and-drop. ROADMAP criterion 1 says "DnD or context-menu merge works". AssetCard has a ContextMenu component already; we add a "Stack onto…" entry that opens a picker modal listing sibling assets in the same folder.

### Gap 6 — VersionStackModal: "current" badge misleads after reorder
`version.id === asset.id` marks "Current" by the asset that opened the modal, not the root. After reorder this can confuse users. Low priority — flagged but not fixed in this plan unless trivial.

### Non-bug (verified): review-link associations
`ReviewLink.assetIds[]` and `Comment.assetId` reference stable asset doc IDs. Stack/unstack/reorder never change an asset's `id` — only `versionGroupId` and `version`. Therefore review-link and comment refs are automatically preserved. **No action required** other than a regression test to confirm.

</audit_findings>

      if (needsReroot) update.versionGroupId = newGroupId;
      batch.update(db.collection('assets').doc(m.id), update);
    });

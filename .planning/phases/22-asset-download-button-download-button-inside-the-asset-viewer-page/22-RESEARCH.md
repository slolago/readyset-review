# Phase 22: asset-download-button - Research

**Researched:** 2026-04-07
**Domain:** Asset viewer UI / GCS signed URL download
**Confidence:** HIGH

## Summary

The asset viewer page (`/projects/[projectId]/assets/[assetId]/page.tsx`) currently has a header with a "Share" button but no download button. All download infrastructure is already built: `generateDownloadSignedUrl` exists in `src/lib/gcs.ts` with `responseDisposition: attachment`, `forceDownload` exists in `src/lib/utils.ts`, and `downloadUrl` is already generated by the bulk assets route (`/api/assets`) and exposed through `AssetCard`. The single-asset API route (`/api/assets/[assetId]`) does NOT currently generate a `downloadUrl` — only a `signedUrl` (inline playback). This is the only gap.

The feature requires two changes: (1) add `downloadUrl` generation to the single-asset GET handler, and (2) add a Download button to the viewer page header that calls `forceDownload` with the `downloadUrl` (or falls back to `signedUrl`).

**Primary recommendation:** Add `downloadUrl` to `/api/assets/[assetId]` GET response, then add a Download button to the viewer header next to the Share button — no new API routes, no new hooks, no new utility functions needed.

## Architecture Patterns

### How Download Works in This Project

**Pattern:** Generate a GCS signed URL server-side with `responseDisposition: attachment; filename="..."`. Client fetches that URL, creates a blob, and programmatically clicks an anchor.

**Two URL types coexist:**
- `signedUrl` — no Content-Disposition, used for inline playback in VideoPlayer/ImageViewer
- `downloadUrl` — `attachment` disposition, used for force-download to disk

**Established data flow (assets list):**
```
/api/assets GET
  → generateDownloadSignedUrl(gcsPath, name)  → asset.downloadUrl
  → generateReadSignedUrl(gcsPath)            → asset.signedUrl
  → Response: { assets: [...] }

AssetCard
  const downloadUrl = (asset as any).downloadUrl
  const signedUrl = (asset as any).signedUrl
  // Download handler:
  const url = downloadUrl ?? signedUrl
  forceDownload(url, asset.name)
```

**Gap for viewer page:**
```
/api/assets/[assetId] GET (current)
  → generateReadSignedUrl(gcsPath, 120) → asset.signedUrl   ✓
  → downloadUrl: NOT generated                              ✗
```

### Anti-Patterns to Avoid
- **Do not create a new API route** for download — just add `downloadUrl` to the existing `/api/assets/[assetId]` GET handler.
- **Do not use `window.open(url)`** — it opens in a new tab instead of downloading; `forceDownload` already handles the correct blob approach with fallback.
- **Do not skip the `downloadUrl` fallback** — if `generateDownloadSignedUrl` throws, fall back to `signedUrl` (same pattern as bulk assets route: `.catch(() => undefined)`).

## Common Pitfalls

### Pitfall 1: useAsset Returns Asset Without downloadUrl
**What goes wrong:** `useAsset` calls `/api/assets/[assetId]` which currently returns only `signedUrl`. The viewer page uses `displayAsset` (either `activeVersion` or `asset`) — neither has `downloadUrl`.
**Why it happens:** The single-asset endpoint was built for playback, not download. The bulk `/api/assets` endpoint was updated in Phase 12/13 for download; the single-asset endpoint was not.
**How to avoid:** Add `generateDownloadSignedUrl` call alongside `generateReadSignedUrl` in `/api/assets/[assetId]` GET. Also generate `downloadUrl` for each version in the versions array (the viewer can switch versions, so all must have download URLs).
**Warning signs:** Download triggers a new tab instead of a disk save — means `downloadUrl` is absent and `signedUrl` fallback is used without `attachment` disposition.

### Pitfall 2: Version Switching — Active Version Has No downloadUrl
**What goes wrong:** User switches to a different version; the active version's `downloadUrl` is absent because only the root asset's URL was generated.
**Why it happens:** The versions array is fetched and signed in the same GET handler. All versions need `downloadUrl` the same way all versions already get `signedUrl`.
**How to avoid:** In the `/api/assets/[assetId]` GET handler, generate `downloadUrl` for each version in the `Promise.all` map, same as `signedUrl`.

### Pitfall 3: GCS CORS / Blob Fetch for Large Files
**What goes wrong:** `forceDownload` fetches the full blob into memory first. For large video files this can fail or feel slow.
**Why it happens:** Cross-origin GCS URLs require blob fetch to trigger `download` attribute behavior.
**How to avoid:** This is a known limitation already accepted in the project (Phase 12 decision). The existing `forceDownload` has a fallback that directly triggers anchor click if fetch fails. No change needed — just use `forceDownload` as-is.

### Pitfall 4: Button Placement Overflow
**What goes wrong:** Adding Download + Share buttons in the header causes overflow on narrow viewports.
**Why it happens:** Header is `justify-between` with `min-w-0` on the left and `flex-shrink-0` on the right.
**How to avoid:** Place Download button inside the right `flex-shrink-0` div, before the Share button. Both buttons are small (`px-3 py-1.5 text-xs`). Only show the Download button when `displayAsset?.status === 'ready'`.

## Code Examples

### Adding downloadUrl to /api/assets/[assetId] GET

```typescript
// Source: src/app/api/assets/[assetId]/route.ts (modify the existing signed URL block)
// Current (line 23-25):
if (asset.gcsPath && asset.status === 'ready') {
  asset.signedUrl = await generateReadSignedUrl(asset.gcsPath, 120);
}

// Replace with:
if (asset.gcsPath && asset.status === 'ready') {
  const [signedUrl, downloadUrl] = await Promise.all([
    generateReadSignedUrl(asset.gcsPath, 120),
    generateDownloadSignedUrl(asset.gcsPath, asset.name, 120).catch(() => undefined),
  ]);
  asset.signedUrl = signedUrl;
  if (downloadUrl) asset.downloadUrl = downloadUrl;
}
```

### Adding downloadUrl to each version in the versions array

```typescript
// Source: src/app/api/assets/[assetId]/route.ts (modify the versions Promise.all, line 45-53)
const versions = await Promise.all(
  versionDocs.map(async (v) => {
    if (v.gcsPath && v.status === 'ready') {
      try {
        const [signedUrl, downloadUrl] = await Promise.all([
          generateReadSignedUrl(v.gcsPath, 120),
          generateDownloadSignedUrl(v.gcsPath, v.name, 120).catch(() => undefined),
        ]);
        v.signedUrl = signedUrl;
        if (downloadUrl) v.downloadUrl = downloadUrl;
      } catch {}
    }
    return v;
  })
);
```

## Open Questions

1. **Should "Download" label be shown or icon-only?**
   - What we know: Share button shows label + icon. Back link shows label + icon.
   - What's unclear: Whether to show text label or icon-only for Download to save space.
   - Recommendation: Show text "Download" to match Share button convention; the header has sufficient space.

2. **Should download work for uploading/pending assets?**
   - What we know: `signedUrl` is only generated when `asset.status === 'ready'`.
   - Recommendation: Hide the Download button when `status !== 'ready'` — consistent with how review links handle it.

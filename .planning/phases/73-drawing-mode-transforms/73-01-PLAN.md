<objective>
Fix DRAW-01: single-object selections in drawing mode expose working Fabric.js scale + rotation handles.

Purpose: The bounding box with corner + rotation handles already renders visually on single-object selection, but dragging them only moves the object. Multi-object selection works because Fabric's `ActiveSelection` wraps objects in a group that is evented by default — but individual objects have `evented = false` set elsewhere in the component and are never restored when switching to the `select` tool. Fabric's hit-testing for control handles requires the underlying object to be evented, so handle drags fall through to the canvas-level move path.

Output: One-line fix in the `'select'` case of the tool-switching `useEffect` (and a matching reset pass) restoring `evented = true` alongside `selectable = true`.
</objective>

Scout results (performed during planning):

- `fabric` imports: ONLY `src/components/viewer/AnnotationCanvas.tsx` uses the fabric library. `ImageViewer.tsx` and `VideoPlayer.tsx` import `AnnotationCanvas` but do not touch the canvas directly.
- `lockScalingX` / `lockScalingY` / `lockRotation` / `lockUniScaling` / `setControlsVisibility` / `hasControls`: **zero matches across the entire codebase.** No lock flags, no controls-hiding. Rules out the "flags were set intentionally" hypothesis.
- Parent wrappers (VideoPlayer line 498, ImageViewer line 149): plain positioned `div` with `pointerEvents: 'all'`. No `onPointerDown` / `onMouseDown` / custom drag handlers. Rules out the "wrapper intercepts handle drags" hypothesis.
- Fabric version: `^5.3.0` (package.json line 30). Controls are on by default at this version.

Root cause — `src/components/viewer/AnnotationCanvas.tsx`:

The tool-switching `useEffect` (lines 137-255) sets every object's `evented = false` on lines 150 (inactive branch) and 156 (pre-tool reset). The `'select'` case at line 162-164 does:
```ts
canvas.selection = true;
canvas.forEachObject((obj: any) => { obj.selectable = true; });
```
It restores `selectable` but NOT `evented`. It also doesn't set `hasControls` (default is already true, so not needed), `lockScalingX/Y`, or `lockRotation`.

Why multi-select still works: `canvas.selection = true` enables Fabric's rubberband group-selection, which creates an `ActiveSelection` (a fresh Group instance) wrapping the selected objects. The `ActiveSelection` wrapper is evented by default — its controls work regardless of the inner objects' `evented` state.

Why single-select breaks: Clicking one object calls `setActiveObject(obj)` directly. Fabric's `__onMouseDown` then hit-tests against the object's own control points. With `evented = false`, hit-testing short-circuits to the canvas-level handler, which treats the drag as a move. The bounding box renders (purely visual; unrelated to events), which is why the UI looks correct but doesn't transform.

Additional note — objects created live (freehand path via `path:created`, rect/circle/arrow, text) get `selectable: false` in their factory (lines 44, 179, 199, 220) but no explicit `evented`. Fabric defaults `evented` to `true` on new objects — BUT line 156 then forces `evented = false` on the next tool switch. So even freshly created objects hit the same bug once the user switches to the select tool.

Fix: in the `'select'` case, set `obj.evented = true` alongside `obj.selectable = true`.
</scouting_notes>
